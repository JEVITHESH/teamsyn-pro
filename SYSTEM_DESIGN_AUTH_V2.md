# System Architecture: Admin-Gated Team Onboarding

## 1. Authentication Flow

### A. Admin Registration
*   **Input**: Email, Password.
*   **Action**: Direct Account Creation.
*   **Role**: `ADMIN`.
*   **Status**: Active immediately.

### B. Team Leader Registration (Request Mode)
*   **Input**: Name, Email, Password, **Requested Team Name**.
*   **Action**: Account Created.
    *   Role: `TEAM_LEADER`.
    *   Status: `Pending`.
    *   Team: `None` (Requested Name stored in profile).
*   **Feedback**: "Account created. Waiting for Administrator approval."

### C. Admin Approval Workflow
1.  Admin logs in.
2.  Admin views **"Pending Team Requests"**.
3.  Admin approves a request:
    *   System creates a new **Team** in the database.
    *   System generates a **Unique Passkey** for this team.
    *   System links the Leader to this new Team.
    *   System activates the Leader's account.
4.  Admin communicates the **Passkey** to the Leader (out-of-band/simulated).

### D. Team Member Registration (Join Mode)
*   **Input**: Name, Email, Password, **Team Passkey**.
*   **Action**:
    *   System verifies Passkey against all active Teams.
    *   If valid: Account Created.
        *   Role: `MEMBER`.
        *   Team: Linked to the matching Team ID.
        *   Status: Active.
    *   If invalid: Registration Blocked.

## 2. Database Structure

### `users` Collection
```json
{
  "id": "uid_123",
  "email": "leader@example.com",
  "role": "TEAM_LEADER",
  "isApproved": false,
  "requestedTeamName": "Alpha Squad", // Only for pending leaders
  "teamId": null // Set after approval
}
```

### `teams` Collection
```json
{
  "id": "team_456",
  "name": "Alpha Squad",
  "passkey": "XYZ-789", // Generated by Admin
  "leaderId": "uid_123",
  "createdAt": 1700000000
}
```

## 3. Security Rules
*   **Isolation**: Members function `getUsers` / `getMessages` checks `auth.uid.teamId`.
*   **Gatekeeping**: No `Team` exists without Admin approval. No `Member` joins without a valid `Passkey`.
